Intrusion Detection and Firewalls

A comprehensive technical overview of Intrusion Detection Systems (IDS), Honeypots, and Firewalls, designed for deep understanding of network security paradigms.

1. Fundamentals of Intrusion Detection

Prevention vs. Detection

In network security, a strong perimeter relies on two major philosophies working in tandem:

  • Prevention: Mechanisms like firewalls and access controls try to stop intrusions before they happen.
  • Detection: An Intrusion Detection System (IDS) is designed to detect an intrusion in progress or after it occurs.
Defense in Depth: Since full prevention is impossible (due to zero-day vulnerabilities, misconfigurations, or insider threats), an IDS gives you the necessary visibility to mitigate damages as early as possible. Furthermore, an IDS acts as a deterrent and pinpoints weaknesses in your primary defenses.

Goals of an IDS

  • Detection & Response: Identify malicious actions instantly to trigger automated or manual mitigation.
  • Deterrence: Increase the perceived risk of an attack for potential perpetrators.
  • Recovery & Analysis: Provide forensic data to understand exactly what the attacker touched.
  • Protection: Defend against future attacks by analyzing vectors used in past attempts.

An IDS detects two main classes of behavior:

  1. External Threats: Illegal access originating from outside the network perimeter.
  2. Internal Threats: Illegal access or privilege escalation from insiders (e.g., an employee accessing restricted financial files).

2. Behavioral Analysis & The False Alarm Tradeoff

The Core Assumption

All IDS systems rely on the key assumption that attacker behavior is measurably different from legitimate user behavior. However, in reality, there is an overlap.

The overlap (labeled XXXXX) represents the area where legitimate behavior looks malicious, or malicious behavior successfully mimics legitimate activity.

False Positives and False Negatives

  • False Positive: The IDS triggers an alarm for completely acceptable behavior. (Also known as a Type I error). Too many false positives lead to alert fatigue, where security teams begin ignoring warnings.
  • False Negative: The IDS fails to identify actual illegal behavior (Also known as a Type II error). This is a critical failure of the security posture.
Example - The Base Rate Fallacy: If you have an IDS that is 99% accurate (i.e. Pr[alarm | attack] = 0.99 and Pr[no alarm | no attack] = 0.99), and the probability of an actual attack on any given day is 1/1000, what is the probability that an alarm indicates a real attack?

Using Bayes' theorem:
Pr[attack | alarm] = (Pr[alarm | attack] * Pr[attack]) / Pr[alarm]
Pr[alarm] = (0.99 * 0.001) + (0.01 * 0.999) = 0.01098
Pr[attack | alarm] = 0.00099 / 0.01098 ≈ 0.09 (or 9%)

This means when the alarm goes off, there's only a ~9% chance it's a real attack—91% of alarms will be false! Improving detection accuracy or combining different IDS signals is essential to lower false alarms.

3. IDS Detection Methodologies

Host-based IDS (HIDS)

A HIDS monitors events on a single host computer. It has deep visibility into system calls, file integrity, and application logs. It can detect internal privilege escalations. HIDS typically employs two approaches:

A. Anomaly Detection

Monitors behavior and compares it against a derived "baseline" of expected normal operations.

  • Threshold Detection: Counting occurrences of events in a time window (e.g., 5 failed logins within 10 seconds).
  • Profile-based (Statistical): Tracking user-specific baselines originally generated from historical audit records. It uses statistical models or Markov chains to detect deviations (e.g., login times, data volume transfers).
Behavioral Metric Model Used Justification / What it Detects
Login frequency (Time/Date) Mean & Std Dev Intruders often operate during off-hours to avoid human detection.
IP Location of Login Mean & Std Dev Detects logins from geographically distant or unusual IPs.
Time since last login Markov (time series) Flags break-ins into dormant, abandoned accounts.
Volume of Data Copied Mean & Std Dev Identifiers attempts to exfiltrate vast amounts of sensitive data.

B. Signature (Rule-based) Detection

Signature-based detection defines a strict set of "bad patterns" (e.g., known malware hashes, bad HTTP headers, buffer overflow payload signatures). If the traffic or file matches the signature, it is blocked.

Heuristic Rule Examples:

  • Users must not write to files in other users' home directories.
  • Users should not directly open disk block devices bypassing higher OS-level abstraction.
  • Users must not be concurrently logged into the system multiple times from different networks.
Key Difference: Anomaly detection discovers atypical behavior (can catch zero-days but has high false positives). Signature detection discovers improper behavior (high accuracy, but blind to brand-new, unknown attacks).

4. Network-Based IDS (NIDS) & Distributed Systems

Unlike HIDS which monitors local activity, a NIDS monitors traffic packet-by-packet at designated choke points within the network.

Sensor Types

  • Inline Sensor: Placed directly in the data flow (like a physical switch). All traffic passes through it. It can instantly drop malicious packets (acting as an IPS - Intrusion Prevention System).
  • Passive Sensor: Monitors a mirrored copy of the traffic (e.g., using a SPAN port on a switch). It is highly efficient and doesn't introduce latency, but it cannot block packets in real-time.

Distributed Architecture Considerations

In massive networks, audit records and traffic statistics are sent from various hosts and network sensors to centralized servers.

  • Centralized: Easy to correlate attacks, but represents a single point of failure.
  • Decentralized: Multiple analysis centers make the IDS more robust but require heavy coordination across the network. Confidentiality and integrity of these internal audit streams are vital so attackers cannot forge or erase logs.

5. Honeypots

A honeypot is an extensively monitored decoy system or network intended to lure, delay, and document attackers. Because it provides no legitimate business value, any connection attempt to a honeypot is inherently suspicious.

Placement Strategies

  • Outside the Firewall: Good for logging global automated port-scanning and brute force attempts. A compromised external honeypot does not risk internal network assets. It will heavily log external scanning attempts.
  • Fully Internal: Placed inside the firewall. It is exceptional at catching rogue insider threats or malware propagating laterally through the trusted network. However, if an attacker completely compromises it, they have a staging ground inside your network.

6. Firewalls

A firewall establishes a central "choke point" for network entry/exit. Its primary goals are service control, behavior control (spam/content filtering), and user-level access restrictions. It may also provide Network Address Translation (NAT) and VPN functionality.

Limitations: Firewalls generally cannot protect against inside attacks, users circumventing via rogue modems/tethering, or devices infected outside the network brought inside (e.g., infected employee laptops).

Types of Firewalls

  1. Packet Filtering: Inspects network layer headers (IPs, Ports, Flags) and applies rules.
    • Pros: Extremely fast.
    • Cons: Ignores context, doesn't look at application data, easily fooled by IP address spoofing, struggles with dynamic ports (like active FTP).

  2. Stateful Inspection Firewalls: Tracks the context and state of active connections. It maintains a state table of TCP connections. Using this context, it can detect replay attacks and easily permit dynamic inbound ports matched with outbound requests.

  3. Application-Level Gateway (Proxy): Acts as a middleman for specific protocols (HTTP, FTP). The connection terminates at the firewall, and the firewall creates a second connection to the target server.
    • Pros: Extremely secure, can deeply inspect payloads (e.g. blocking specific URLs or malformed SQL).
    • Cons: High processing overhead; if a new application isn't supported by the proxy, it simply can't pass.

  4. Circuit-Level Gateway: Operates at the transport layer (TCP/UDP). Similar to an application gateway, it setups two TCP connections but does not examine the payload contents once the handshake is authenticated. It purely ensures the session is valid.

VPNs and IPsec

Firewalls are often the endpoints for Virtual Private Networks using IPSec. While standard firewalls cannot analyze the encrypted payload of an IPSec packet, they encrypt the "plain IP packet" into a "secure IP packet," enabling secure bridging of two separated local networks over the public Internet.

7. Practice Questions

Test your understanding of the concepts across three difficulties.

EASY
Q1: What is the primary difference between prevention and detection in network security?

A: Prevention (e.g., Firewalls) aims to actively stop an intrusion before it penetrates the perimeter. Detection (e.g., IDS) aims to identify an intrusion that is currently happening or has already bypassed preventative barriers.

EASY
Q2: Why does an IDS trigger "false positives"?

A: Because there is an overlap between legitimate user behavior and malicious behavior. If a legitimate user acts in an atypical way (like mistyping their password 5 times), the IDS interprets it as a brute-force attack.

MEDIUM
Q3: Compare and contrast Signature-based detection and Anomaly-based detection. Explain a scenario where one would fail while the other succeeds.

A: Signature detection looks for known bad patterns (improper behavior) and uses a database of known threats. Anomaly detection establishes a baseline of normal activity and flags deviations (atypical behavior).

Scenario: A brand-new zero-day vulnerability is released. Signature detection will completely ignore it because there is no matching signature in its database yet. Anomaly detection, however, might catch it if the exploit causes the server to suddenly transmit strange data patterns or spike the CPU abnormally.

MEDIUM
Q4: What is the difference between an Application-level gateway and a Circuit-level gateway?

A: Both act as proxies connecting client to server via two separate TCP connections. However, an Application-level gateway deeply inspects the actual application payload (e.g., ensuring an HTTP packet contains a valid GET request). A Circuit-level gateway only establishes and authenticates the TCP session. Once established, it forwards TCP segments blindly without looking at the application data.

HARD
Q5: Using Bayes Theorem, prove why high-accuracy Intrusion Detection Systems generate a massive amount of false alarms in real-world scenarios. Explain what variables a security engineer must tune to fix this.

A: Real-world networks have relatively few actual attacks compared to the colossal number of normal packets (the Base Rate Fallacy). Even if an IDS is 99% accurate at detecting attacks and 99% accurate at identifying non-attacks, if actual attacks only make up 0.1% of network events, the vast majority of alarms will be triggered by the 1% error rate on the 99.9% normal traffic. The resulting Pr[attack | alarm] becomes drastically low (e.g. 10%).

To fix this, an engineer cannot change the probability of an attack happening (Pr[attack]). Therefore, they must radically decrease the false positive rate (i.e. increase Pr[no alarm | no attack] to 99.9999%) or use contextual data outside the mathematical model to discard irrelevant alerts.

HARD
Q6: A corporate network deploys a honeypot entirely inside the trusted LAN. What are the specific architectural benefits and the potentially catastrophic risks? How does a stateful firewall placement affect this?

A: The benefit of an internal honeypot is high-fidelity detection of insider threats, lateral malware propagation, and firewall misconfigurations. Since no legitimate internal user should ever access it, alerts are extremely reliable.

The catastrophic risk is that if the attacker uncovers a vulnerability on the honeypot itself (e.g. escaping the sandbox or exploiting a service running on the decoy OS), the attacker now has a fully compromised, trusted pivot point directly on the internal network. A stateful firewall cannot help mitigate this pivot if internal subnet routing allows any-to-any connections internally. The honeypot must reside on a heavily isolated internal VLAN with strict egress filtering to prevent it from becoming a launchpad.
